# Comments

Write comments only when they add understanding the code itself doesn't convey —
a non-obvious **why** (a constraint, a trade-off, a gotcha). If the names and
structure already make the intent clear, write no comment.

- Don't restate what the code does, narrate steps, or decorate sections.
- Skip `///` dartdoc when the class/function name already conveys intent; reserve it for non-obvious behavior.
- Keep comments concise — one or two lines.
- When editing existing code, remove redundant comments you come across in the region you touch.
